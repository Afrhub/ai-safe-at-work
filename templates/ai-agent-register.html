AI Agent Register for [Company Name]

Register every AI agent that can take actions on its own, calling tools, APIs and MCP servers, browsing, or transacting. Record what each agent is authorised to do, its autonomy level, its human-in-the-loop checkpoints, and how to stop it. Aligns with EU AI Act Art 4 (AI literacy), Art 14 (human oversight), Art 26 (deployer obligations), and ISO/IEC 42001 Annex A operational controls.

Maintain this register in a shared spreadsheet. Review monthly. Update the same day an agent's tools, permissions, or autonomy level change.
Purpose and scope

Why we register agents

An AI agent is not just a chatbot, it can take actions on your systems and on the outside world: call tools and APIs, browse the web, read and write data, send messages, move money, or connect to MCP servers. An AI Agent Register records what each agent is, what it may do, who is accountable for it, and how a human can stop it. It is:

  • Control, makes explicit what actions each agent is permitted to take and what it is forbidden to do.
  • Oversight, documents the human-in-the-loop checkpoints and kill-switch required by EU AI Act Article 14.
  • Accountability, names a single accountable human for every autonomous system (Art 26 deployer obligations).
  • Audit evidence, shows that autonomy is bounded, logged, and reversible, key to demonstrating governance maturity under ISO/IEC 42001 Annex A.

Re-assessment trigger: if an agent is given new tools, wider permissions, a higher autonomy level, or is repurposed so that it materially changes what it does, treat it as a new deployment, re-run the risk assessment and re-approve before the change goes live.

Scope (what counts as an agent)

This register covers any AI system that can act, not just answer. Register it here if it:

  1. Calls tools, functions, APIs, or MCP servers (e.g. an assistant that can create tickets, send email, or query a database).
  2. Browses or operates software autonomously (e.g. a computer-use or browser agent that clicks and fills forms).
  3. Transacts or changes state (e.g. places orders, refunds customers, updates records, deploys code).
  4. Runs on a schedule or a trigger without a human prompting each action (e.g. a nightly reconciliation agent, an inbox-triage agent).
  5. Chains multiple steps or sub-agents to complete a task with limited or no human intervention.

A plain chat tool with no ability to take actions belongs in the AI Tool Register, not here. If it can act, it belongs here.

Who maintains this register

Ownership: [AI governance lead / Head of Engineering / DPO]

Input sources: engineering deployment records, MCP server configurations, integration and API-key audits, procurement requests, incident reports, vendor change notifications.

Review cadence: Monthly. Update the register the same day an agent's tools, permitted actions, data access, autonomy level, or spend limits change, and re-run the risk assessment before any material change goes live.

Register fields explained

Agent name & version

The name staff and engineers use for the agent, plus a version or build reference so you can tell which behaviour is live (e.g. "Invoice-Triage Agent v2.3"). Behaviour changes between versions, pin the version.

Owner / accountable human

One named person accountable for the agent's behaviour, not a team inbox. This is the human who answers for what the agent does and who can authorise its shutdown (EU AI Act Art 26).

Purpose & authorised task

The specific job the agent is authorised to do, in plain English. Be narrow: "reconcile supplier invoices against purchase orders and flag mismatches", not "help with finance". Anything outside this is out of scope and triggers re-assessment.

Autonomy level

How much the agent may do without a human. Choose one:

  • Suggest-only: the agent proposes; a human takes every action. Lowest risk.
  • Act-with-approval: the agent prepares an action but a human must approve it before it executes (human-in-the-loop).
  • Act-autonomously: the agent executes without per-action approval, within its bounded permissions. Highest risk, requires the strongest oversight, limits, and logging.

Human-in-the-loop checkpoints (Art 14)

The specific points at which a human must review, approve, or can intervene, and who that human is. EU AI Act Article 14 requires that oversight is effective: the person must be able to understand the agent's output, decide not to use it, and override or stop it. Name the checkpoints (e.g. "any payment over £500 requires finance approval").

Kill-switch / how to stop it

The concrete way to stop the agent immediately, and who can trigger it (e.g. "disable API key in vault", "toggle feature flag [flag-name]", "revoke MCP server token"). If there is no fast way to stop it, that is a finding, fix it before go-live.

How to complete and maintain this register

  1. Find the agents: ask engineering and IT what runs on a schedule or has tool/API/MCP access. Audit API keys, service accounts, and MCP server configs. List every agent that can take an action.
  2. Bound each agent: for every agent, write down its narrow authorised task, its permitted and forbidden actions, and its data read/write scope. If any of these is vague, tighten it before approval.
  3. Set oversight and limits: assign one accountable human, define the autonomy level, the Article 14 checkpoints, the kill-switch, and the spend/rate limits. Confirm logging is wired to a retrievable audit trail.
  4. Approve and link: link the agent to its AI System Card and current risk assessment. Do not approve act-autonomously agents without a working kill-switch and audit logging.
  5. Maintain monthly: review every agent. Any new tool, wider permission, higher autonomy level, or change of purpose is a material change, re-run the risk assessment and re-approve before it goes live. Update the register the same day.

Standards referenced

  • EU AI Act 2024/1689, Article 4 (AI literacy), Article 14 (human oversight, including the ability to intervene and stop the system), Article 26 (deployer obligations, including monitoring and named accountability).
  • ISO/IEC 42001:2023, Annex A operational controls (operational planning and control of AI systems, including monitoring, logging, and control of changes).
  • ISO/IEC 27001:2022, Annex A.8.16 (monitoring activities), A.5.15 / A.8.2 (access control and privileged access) for the tools and credentials agents are granted.

Rolling this out
  1. Set up the register in a shared spreadsheet (Google Sheets, Excel, Notion, etc.). Copy the full field set above.
  2. Ask engineering and IT: "What agents can take actions here, on a schedule, a trigger, or with tool/API/MCP access?" Audit keys and MCP configs.
  3. For each agent, bound its task, permitted vs forbidden actions, data scope, autonomy level, checkpoints, kill-switch, and limits.
  4. Link each agent to its AI System Card and risk assessment. Block any autonomous agent that lacks a kill-switch or audit logging.
  5. Assign one owner to review monthly and re-assess on any material change (new tools, wider permissions, higher autonomy, new purpose).
