Governing Microsoft Copilot.

Copilot for M365 is the default-on AI for hundreds of millions of staff. The governance problem isn't "should we use it", it's "what does it surface to whom, and what is logged". This track answers both, and the dozen things in between.

By the end of this track, you'll
  • Tell apart Copilot for M365, Copilot Studio, and Copilot for Sales / Service / Finance, and the governance differences
  • Understand the permissions-inheritance problem and why most Copilot incidents trace back to it
  • Know where prompts and responses live, tenant region, Bing grounding, Azure OpenAI region, non-grounding settings
  • Operate Purview sensitivity labels so Copilot respects them, and verify the operation works
  • Govern Copilot Studio agents without accidentally building an exfiltration tool
  • Audit who used what when, where the logs are, and how long they're kept
  • Recognise risky usage signals via Defender for Cloud Apps
  • Run a Copilot data-exposure incident in the first hour with the right people in the right order
  • Pass a 10-question final assessment that exercises the M365 governance levers

"Copilot" is now a brand, not a product. Five materially-different things share the name. The governance posture differs at every one. Get the naming right or you'll govern the wrong surface.

Five different products share the Copilot brand. Govern the one you actually have, not the one in the marketing video.
Sub-module 2 of 12

The permissions-inheritance problem

The single most common Copilot incident: an end user types a benign prompt, and Copilot returns a perfect summary that includes data the user could technically see but had no idea existed. Salary spreadsheets in a forgotten SharePoint library. Board minutes shared "with everyone in the org" three years ago. Performance reviews in a Team channel nobody removed them from.

Copilot does not break permissions. It surfaces what the user already had permission to see. The problem is that "had permission" is not the same as "knew about". Most M365 tenants have years of accreted over-broad permissions.

Why this is acute for Copilot

  • SharePoint inheritance defaults to "everyone in the organisation" on many site templates.
  • Microsoft 365 Groups create site collections that staff don't realise are "open".
  • Shared links sent in chat persist indefinitely unless deliberately rotated.
  • Files that were "shared with the team" four reorgs ago retain those permissions.
  • Pre-Copilot, retrieving a forgotten file required an explicit search by name. Copilot retrieves it via a natural-language question the user didn't know would match.

What to do before turning Copilot on

  1. Run SharePoint Advanced Management's oversharing report across the tenant. Filter for sites with "everyone except external users" or "all employees" group permissions.
  2. Re-permission sensitive site collections (HR, Finance, Legal, M&A, board) to membership-only.
  3. Apply Purview Information Protection sensitivity labels (Sub-module 4) to sensitive files; Copilot honours labels.
  4. Run SharePoint site-access reviews on a recurring schedule. Treat the first one as a project, not a click.
  5. Pilot Copilot to a small group first; sample-audit the answers it returns for over-share.
!
Restricted SharePoint Search ≠ governance

"Restricted SharePoint Search" limits what Copilot reads to a curated allowlist of sites. It is a stopgap during cleanup, not a long-term posture. Plan for full posture once permissions are right.

Sub-module 2 takeaway
Copilot doesn't surface what people couldn't see. It surfaces what they could see but forgot they could see. Clean the permissions before turning the volume up.
Sub-module 3 of 12

Data residency, where Copilot prompts and responses live

Five separate location settings combine to define "where your data is" in a Copilot interaction. They are not all the same setting; they interact; some are tenant-set and some are user-context-set. DPOs asked "where is the data" want to walk all five.

The five levers

  1. Microsoft 365 tenant region, set at tenancy creation; defines where tenant data is at rest.
  2. Multi-Geo capabilities, optional add-on for splitting workloads across geographies (EU + UK + US within one tenant).
  3. Microsoft 365 Copilot processing, happens within the Microsoft 365 service boundary; the model is hosted in Azure infrastructure with regional alignment per Microsoft's published service-boundary documentation.
  4. Bing grounding (web search), when Copilot calls out to the public web for current information. Tenant admin can disable this (`AllowWebSearch` setting + tenant-level controls).
  5. Copilot Studio agents calling external endpoints, actions, Power Automate flows, Dataverse plugins. Each external call is a residency event in itself.

The questions your DPO will ask

  • "Does our tenant region cover this user's data?", check Multi-Geo configuration; some users may have data in a different geo.
  • "Are we using Bing grounding?", if yes, public-web data is in the loop; if no, document the disable.
  • "What is our EU Data Boundary commitment?", Microsoft offers an EU Data Boundary commitment for tenants with EU billing addresses; confirm it applies to Copilot (it does, with specific service-coverage caveats Microsoft publishes).
  • "Is Copilot using customer content to train Microsoft's foundation models?", Microsoft's published terms: no. Confirm against the current Service Terms version.
  • "How do we evidence this in an audit?", Microsoft Service Trust Portal compliance documents; tenant settings export; sample tenant audit logs.
§
EU Data Boundary, confirm not assume

Microsoft publishes which services are in scope for the EU Data Boundary commitment and which are not yet. Copilot scope evolves quarterly. Check the current Service Trust Portal documentation against your audit cycle. Don't reuse last year's answer.

Sub-module 3 takeaway
Five levers, not one. A DPO who asks "where is the data" deserves a five-line answer, not a wave at "the tenant".
Sub-module 4 of 12

Sensitivity labels and Purview Information Protection

Microsoft Purview Information Protection lets you label content (Public / Internal / Confidential / Highly Confidential / custom) and apply protection that Copilot respects. Used well, labels are the strongest preventive control against Copilot over-exposure. Used badly, they're shelfware.

What labels do

  • Visual marking, header / footer / watermark on documents.
  • Encryption, files inherit RMS-style encryption with rights tied to identity.
  • Access control, only specified users / groups can decrypt.
  • Copilot behaviour, when a labelled file's protection would block read, Copilot does not surface or summarise it for the user.
  • Audit, label apply / modify / remove events are captured in the unified audit log.

Practical labelling rollout

  1. Start small, three labels, not seven. Public / Internal / Confidential.
  2. Auto-apply for high-confidence patterns (credit card numbers via Purview's sensitive-info types; HR keywords in HR site collections).
  3. Recommend (not enforce) for ambiguous cases, let users confirm.
  4. Run a label-coverage dashboard in Purview; track % of relevant content labelled.
  5. Validate Copilot honours the label by attempting (as test user) to ask Copilot about a labelled-Confidential file the test user can't decrypt. Expect: no surface.
!
Labels without protection are weak

A "Confidential" label that just adds a watermark and no encryption is mostly cosmetic. Pair every Confidential-or-above label with encryption + access scope. The label name on its own does not constrain Copilot reads.

Sub-module 4 takeaway
Sensitivity labels are the lever that makes Copilot governance enforceable. Three labels with protection beat seven labels with watermarks.
Sub-module 5 of 12

Copilot Studio, governed agents (or accidental exfiltration)

Copilot Studio lets non-developers build agents. The control surfaces are good, but the defaults are permissive, designed for adoption, not governance. Governance leads who don't watch Copilot Studio find a fleet of citizen-built agents running on tenant data within months.

What can go wrong

  • An agent connects to a SharePoint site via an authenticated connector, using the maker's identity, not the end user's. Anyone who chats with the agent sees what the maker can see, including things they could not see directly.
  • An agent calls an external HTTP endpoint via a Power Automate flow, sending tenant data outside the boundary.
  • An agent uses generative answers grounded on a public website your team thought was internal.
  • An agent ships into a Team channel without DLP review.
  • The maker leaves the company; their agent keeps running on their credentials until it breaks.

Governance controls

  1. Environment strategy, separate Default / Dev / Prod environments; restrict who can publish to Prod.
  2. DLP policies, categorise connectors as Business / Non-business / Blocked; prevent mixing.
  3. Maker governance, restrict who can be a maker; require Power Platform admin approval for new makers.
  4. Tenant settings, disable "anyone-in-the-org" publishing of Copilot Studio agents until reviewed.
  5. Audit, Power Platform audit logs surface agent runs, connector calls, and external endpoint hits.
  6. Lifecycle, agents must have an owner; ownership transfer on leavers must be enforced via Power Platform admin checks.
§
Citizen-builder review board

Established M365 tenants typically benefit from a monthly "agent review board", IT + DPO + a citizen-builder rep meet for 30 minutes, walk new agents that touched sensitive data or external endpoints, approve / require-changes / retire. This catches drift before it becomes a discovery exercise.

Sub-module 5 takeaway
Copilot Studio agents look like apps and have the governance needs of apps. Treat them as such, owners, lifecycle, DLP, environment hygiene, or accept a citizen-built shadow IT estate.
Sub-module 6 of 12

Microsoft 365 group hygiene, the upstream control

Microsoft 365 Groups are the unit of permission for most M365 surfaces, Teams, SharePoint sites, Planner, Outlook groups, Yammer / Viva communities. Most of what Copilot can see flows from group membership. Bad groups → bad Copilot answers.

Hygiene tasks that move the needle

  1. Naming policy, enforce prefix conventions per business unit; rejects accidental Sales-named groups owned by Finance.
  2. Expiration policy, groups expire if not actively used; owners must renew.
  3. Owner requirement, every group has at least two owners; orphan-owner cleanup is a recurring job.
  4. Access reviews, Entra ID Identity Governance access reviews on a quarterly cadence for sensitive groups.
  5. Guest user audit, external guests in a group should have a documented purpose + expiry; "left the project two years ago, still in the channel" is the most common case.
  6. Dynamic membership, for stable groups (e.g. "all UK employees"), dynamic membership rules in Entra ID prevent drift.
!
"Everyone except external users" is a footgun

This group resolves to every internal employee. Permissions granted to it ripple to every Copilot user. Avoid using it for sensitive content; prefer explicit groups, even when membership matches.

Sub-module 6 takeaway
Copilot governance is downstream of Entra ID governance. Clean groups → clean Copilot. Don't try to govern Copilot while leaving the groups dirty.
Sub-module 7 of 12

End-user prompt patterns, what to ask, what to verify

Even with permissions clean and labels right, the end user holds the last gate. Copilot will produce confident-sounding summaries even when the underlying data is partial, stale, or hallucinated. Prompt patterns + verification habits are the user-level control.

Patterns that work

  • Cite-as-you-summarise, ask Copilot to include file references in any summary; spot-check at least one citation per summary.
  • Boundary-the-source, "Using only files in this Teams channel, summarise…" beats "summarise our Q3 results" (which may pull from outdated, unrelated documents).
  • Two-step verify, for any output that goes externally (customer email, board paper, regulator response): ask Copilot to draft, then verify each substantive claim against the cited source by hand.
  • Sensitive-topic redirect, for HR / disciplinary / legal / financial questions, redirect to the named owner of those records, not to Copilot.

Patterns that don't

  • "Summarise everything about Project Phoenix", too broad; Copilot pulls in old emails, stale drafts, irrelevant chats.
  • "Find the contract", depending on permissions, may surface multiple unrelated contracts; ambiguous.
  • "Write a customer apology", without verifying the underlying facts the apology becomes an admission of something that may not be true.
  • "How much was paid in commission last quarter", Copilot may calculate from partial data and present as authoritative.
Sub-module 7 takeaway
The user holds the last gate. Cite, bound, verify. Confident summaries are not verified summaries.
Sub-module 8 of 12

Audit logging in Microsoft 365

Microsoft 365 captures Copilot interactions in the unified audit log. What's captured + retention period depends on licence tier. Get this wrong and you can't reconstruct what happened during an investigation.

What is captured

  • The user, the timestamp, the application (Word / Outlook / Teams / SharePoint / Copilot chat).
  • The prompt text and the response (depending on tenant settings and licence).
  • Files referenced in the grounding context.
  • Whether Bing grounding was used.
  • Labels applied to surfaced content.

Retention defaults, confirm against your licence

  • Standard audit retention is 180 days for most events on most enterprise licences. Verify against current Microsoft documentation for your specific licence (E3 / E5 / Business Premium have different defaults).
  • EU AI Act Article 26(6) requires deployers of high-risk AI systems to keep auto-generated logs for at least six months; longer if other law applies.
  • Standard GDPR rules apply to audit logs containing personal data; minimisation and retention review needed.
  • For longer retention, configure a Purview retention policy or export to a SIEM (Sentinel, Splunk, etc.).

What you should query

  • "What did user X ask Copilot in the week before they handed in their notice?", leaver risk.
  • "Which Copilot interactions referenced files labelled Confidential or higher?", sensitivity scrutiny.
  • "Did anyone ask Copilot about Project Phoenix outside the project team?", info-leak detection.
  • "What % of Copilot interactions in HR contexts used grounded vs Bing-grounded responses?", boundary integrity.
Sub-module 8 takeaway
If you can't answer "what did this user ask Copilot last Tuesday", you don't have audit. Default retention runs out before most investigations start, extend it explicitly.
Sub-module 9 of 12

Defender for Cloud Apps, detecting risky Copilot usage

Microsoft Defender for Cloud Apps (MDCA, formerly MCAS) is the user-and-entity-behaviour layer that surfaces patterns the unified audit log alone won't. For Copilot governance, MDCA gives you the signals + alerts.

Signals worth alerting on

  • Unusual volume of Copilot prompts from a single user (especially around leaving / dispute periods).
  • Copilot interactions from atypical countries or unmanaged devices.
  • Copilot prompts referencing sensitive-info types Purview has flagged.
  • Patterns that look like systematic data harvesting (sequential prompts walking a folder structure).
  • Mass-download events following Copilot summary outputs.
  • Use of Copilot Studio agents outside their intended environment.

How to operationalise

  1. Enable MDCA's Microsoft 365 connector if not already.
  2. Create Copilot-specific policies via MDCA's app governance + activity policy templates.
  3. Route alerts to your SIEM and to the named Copilot-incident owner.
  4. Review false-positive rates weekly during initial tuning; monthly after.
  5. Cross-reference MDCA alerts with Purview DLP alerts, many incidents trip both.
Sub-module 9 takeaway
MDCA is the lens. Tune it for Copilot signals, route the alerts, review them weekly until tuning is right. Default templates are a starting point, not a finished policy.
Sub-module 10 of 12

Incident response for a Copilot data-exposure event

A Copilot exposure is usually one of: (a) user-summary surfaces data the user shouldn't have seen; (b) user-summary surfaces data correctly, but user emails it to wrong recipient; (c) Copilot Studio agent leaks data via external connector. Same incident form, see the incident report template, but the first-hour choreography is Copilot-specific.

First hour, Copilot variant

  1. 0–5 min, Confirm the variant (Copilot for M365 / Studio / Sales etc.); preserve the chat thread (export to PDF before any user touches the chat); tag the audit log query.
  2. 5–15 min, Identify the file(s) surfaced via the unified audit log + Copilot interaction record. Determine whether the user should have had access to those files via SharePoint / Teams / OneDrive permissions.
  3. 15–30 min, Contain: revoke the user's Copilot access if intentional misuse; revoke the permissions on the surfaced files if a misconfiguration; quarantine the surfaced files if extraction is suspected.
  4. 30–45 min, Sweep: check MDCA for similar patterns from other users; check Purview DLP for related events; identify whether the underlying permissions issue is one file or a system-wide problem.
  5. 45–60 min, Decision point: GDPR Article 33 personal data exposure? Trigger the 72-hour clock. Open the incident-form workflow.

Common containment errors

  • Telling the user to "delete the chat", this destroys evidence and doesn't roll back the exposure. Preserve, don't delete.
  • Revoking the user's Copilot licence without considering whether the underlying permissions are the actual problem, the next user will hit the same exposure.
  • Treating this as "a Copilot problem" rather than a permissions / labelling problem, fix the upstream cause, not the surface.
Sub-module 10 takeaway
Copilot doesn't cause the exposure; it reveals the permissions problem. Preserve evidence, fix the root cause, communicate up-front. The chat is evidence, not embarrassment to delete.

Sub-module 12 of 12

Final assessment

Ten questions exercising the M365 governance levers. Self-mark. Suggested pass: 8/10.

Question 1

A staff member asks "show me everything about our pay rises" in M365 Copilot. The summary includes the leadership team's bonus figures. The user had no idea those documents existed. Whose problem is this?

Answer

The permissions are the problem, not Copilot. The leadership-bonus document was probably stored in a SharePoint site with over-broad inheritance or in a group the user was a member of for an unrelated reason. Fix: audit + relabel + restrict; Copilot then behaves correctly. Treating it as "a Copilot incident" obscures the real cause.

Question 2

Your DPO asks: "Where does our Copilot prompt data live?" Give the one-paragraph answer.

Answer

Five levers determine residency: (1) M365 tenant region (set at tenant creation); (2) Multi-Geo configuration (if used); (3) Copilot processing happens inside Microsoft's M365 service boundary aligned with tenant region; (4) Bing grounding (configurable on/off) introduces public-web data when enabled; (5) Copilot Studio agents calling external endpoints can move data outside the boundary per-action. EU Data Boundary commitment covers most Copilot scope for EU-billed tenants, confirm via Service Trust Portal.

Question 3

A team labels a folder "Confidential" using a Purview sensitivity label, but they don't apply encryption with the label. A user asks Copilot to summarise the folder contents. Does Copilot honour the label?

Answer

Mostly no, Copilot honours the protection (encryption + rights) tied to the label, not the visual label name. A "Confidential" label without encryption gives watermarks but doesn't prevent Copilot reads. Fix: pair every Confidential-or-above label with encryption + access scope.

Question 4

Citizen builder publishes a Copilot Studio agent that helps "anyone in finance" summarise contracts. Three months later the agent is used by someone in sales because the publish scope was wide. What broke?

Answer

The Copilot Studio environment + publish-scope governance broke. Fixes: environment separation (Dev / Prod), DLP categorisation of connectors, restriction of who can publish to Prod, tenant setting limiting "anyone-in-org" publish scope, mandatory ownership + lifecycle reviews. Citizen builders aren't the problem, un-governed citizen building is.

Question 5

You want to know what user X asked Copilot last Tuesday. You query the M365 unified audit log; you find nothing. What's wrong?

Answer

Several possibilities, often combined: (a) audit retention default has lapsed; (b) Copilot-specific audit settings weren't fully enabled; (c) the user's licence tier doesn't include detailed audit; (d) prompts and responses retention requires explicit tenant configuration. Fixes: extend audit retention via Purview retention policy or export to SIEM; verify M365 Copilot audit settings; confirm licence covers the audit detail you need.

Question 6

Microsoft updates the underlying Copilot model. Output behaviour shifts. Is this a change-management event?

Answer

Yes, ISO 27001 A.8.32 (change management) and ISO 42001 A.6.2.5 (AI system deployment) both apply. Document the change; re-test critical workflows; communicate to dependent customers if you are an MSP; update the approved-tool inventory.

Question 7

An MDCA alert fires: "user X executed 47 Copilot prompts in the hour before logging out for the last time as a leaver". What do you do?

Answer

(1) Preserve the chat threads + audit records before any user touches them. (2) Identify what was surfaced, files, labels, sensitive content. (3) Engage HR + legal + the leaver's manager. (4) Determine whether this constitutes a data-exfil event requiring incident escalation. (5) If yes, open incident-form workflow + start GDPR Art 33 clock if personal data involved.

Question 8

You are running an SMB without dedicated security staff. You can do one of three things tomorrow before turning Copilot on. Pick.

  1. Buy SharePoint Advanced Management and run the oversharing report
  2. Deploy Purview sensitivity labels across the tenant
  3. Set up Defender for Cloud Apps with all the templates enabled
Answer

A. The oversharing report identifies the upstream permissions problem that Copilot will surface. Labels (B) help but only on content that's already in scope; MDCA (C) detects after-the-fact. Permissions cleanup before turning Copilot on prevents the largest class of incident.

Question 9

Marketing wants Copilot to draft customer emails. HR wants Copilot to summarise candidate CVs. Which is fine, which needs more thought?

Answer

Marketing emails generally fine with cite-as-you-summarise verification (Module 5 / Sub-module 7). HR CV summarisation may engage GDPR Article 22 (automated decisions affecting individuals) and the EU AI Act's higher-risk category for employment-related AI use. Don't ship CV summarisation without DPO sign-off, a documented oversight mechanism, and likely a DPIA. The Manager Portal and FRIA template cover the additional duties.

Question 10

You're an MSP. A customer asks you to "set Copilot up". What do you do first?

Answer

Set up the governance posture before the licence, see the MSP / IT Admin track sub-module 5 (customer onboarding sequence). Specifically: oversharing report + sensitivity-label baseline + group-hygiene audit + audit-log retention configuration + AUP signed by their staff + their training register opened. Then turn Copilot on to a pilot cohort, sample-audit, expand. Selling "Copilot turned on" without governance produces incidents that come back to you.
