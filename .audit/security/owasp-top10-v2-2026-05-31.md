# OWASP Top 10 (2021) — v2 platform review

> Date: 2026-05-31
> Scope: every page under `/v2/*` of the AI Safe@Work platform.
> Auditor: Founder (self-audit).
> Posture: **STATIC HTML site. No server-side code. No database. No API.** This shrinks the OWASP attack surface dramatically; the live ones are bot, supply-chain and storage-API issues, not server-side issues. Each item below is documented anyway for procurement-readiness.

---

## Architecture summary (the threat model)

| Layer | Technology | Trust |
|---|---|---|
| Hosting | Netlify (today) → Cloudflare Pages (planned) | Vendor-trusted; site delivered over HTTPS w/ HSTS preload |
| Code | Plain HTML + 1 CSS file + 1 JS file. No build pipeline. No framework. | Hand-written; reviewed in this audit |
| Data | `modules.json` (static, server-side same-origin) + browser `localStorage` (client-only) | No PII collected; nothing leaves the device |
| Third-party | Google Fonts (CSS + woff2). Read-only. No third-party JS. | Vendor-trusted; CSP restricts what they can do |
| Auth | None | N/A |
| API | None | N/A |
| DB | None | N/A |

## OWASP A01 — Broken access control

**Applicability:** LOW. No accounts, no roles, no protected resources. Every page is public.

**Mitigations in place:**

- No admin paths exist. `_headers` blocks `/admin/*` if one is ever introduced (sends `X-Robots-Tag noai, noimageai, nosnippet, noindex, nofollow` + `Cache-Control: private, no-store`).
- `frame-ancestors 'none'` in the v2 CSP prevents clickjacking via iframe embedding.
- `.audit/` directory blocked via `netlify.toml` + `_headers`.

**Residual risk:** Negligible.

---

## OWASP A02 — Cryptographic failures

**Applicability:** LOW. No credentials handled. No tokens issued. No encrypted data at rest.

**Mitigations in place:**

- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` — TLS enforced across all paths.
- `upgrade-insecure-requests` directive in CSP.
- No localStorage entry contains PII or secrets (only module-progress integers and a UI-dismissed flag).

**Residual risk:** Negligible.

---

## OWASP A03 — Injection

**Applicability:** MEDIUM (XSS only — there is no SQL, OS or LDAP layer to inject into).

**Mitigations in place:**

- All dynamic content in `assets/v2.js` uses `document.createTextNode()` and `setAttribute()`. **No `innerHTML = ` of user-derived strings.** (Static markup uses `innerHTML = ''` only to clear a container before appending textNodes.)
- `modules.json` is read from same-origin only. Each entry is **shape-validated** (regex on `file`, type check on `id`, `title`) before use.
- `localStorage` reads are **shape-validated** before use (`/^\d+$/` on keys, enum check on values). Malformed entries are dropped.
- `href` attribute injection is **blocked at the helper level** — `el()` rejects any href starting with `https?:`, `javascript:`, or `data:`. Only same-origin relative paths get through.
- CSP `default-src 'none'` + `script-src 'self'` blocks inline script execution and any externally-loaded script.
- No `eval`, no `Function()`, no `setTimeout(stringArg)`, no `setInterval(stringArg)` anywhere in `v2.js`.

**Verification commands:**

```bash
# Should return ZERO hits in v2.js
grep -nE "innerHTML\s*=|eval\(|new Function\(|setTimeout\(\s*['\"]" v2/assets/v2.js
```

**Residual risk:** Low. Attacker would need to compromise the same-origin `modules.json` to inject content; if they can do that, they already own the host.

## OWASP A04 — Insecure design

**Applicability:** LOW.

**Mitigations in place:**

- Threat model documented above. Each external dependency named.
- All data flows are one-way: server → browser → localStorage. No round-trip, no exfil channel.
- "Mark complete" is a single-state idempotent toggle (no decrement path that could underflow). Throttle on localStorage writes prevents runaway storage.
- Manifest is a single source of truth — change one JSON entry to add, remove or reorder a module. No code change required for content edits. Reduces risk of stale hardcoded references.

**Residual risk:** Negligible.

## OWASP A05 — Security misconfiguration

**Applicability:** MEDIUM (a misconfigured header is the most likely real vulnerability for a static site).

**Mitigations in place — `_headers` block for `/v2/*`:**

| Header | Value | Why |
|---|---|---|
| `Content-Security-Policy` | `default-src 'none'; script-src 'self'; style-src 'self' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self'; manifest-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'none'; object-src 'none'; upgrade-insecure-requests` | Default-deny posture. Only same-origin scripts, Google Fonts CSS+woff2, same-origin XHR (for `modules.json`), no frames, no plugins, no forms. |
| `X-Robots-Tag` | `noindex, nofollow` | v2 is MVP-trial; do not index until founder approves promotion to canonical. |
| `X-Frame-Options` | `DENY` | Belt-and-braces alongside `frame-ancestors 'none'`. |
| `X-Content-Type-Options` | `nosniff` | Stop MIME-sniffing. |
| `Referrer-Policy` | `no-referrer` | Stronger than the site-wide `strict-origin-when-cross-origin` because v2 has no need to send any referrer anywhere. |
| `Permissions-Policy` | `accelerometer=(), camera=(), geolocation=(), gyroscope=(), magnetometer=(), microphone=(), payment=(), usb=(), interest-cohort=()` | Disable every powerful API. v2 needs none of them. |
| `Cache-Control` | `public, max-age=300, must-revalidate` | Short cache; allow rapid content fixes. |
| `Strict-Transport-Security` | inherited site-wide (`max-age=63072000; includeSubDomains; preload`) | HTTPS enforced. |

**Verification:**

```bash
# After deploy, check headers on /v2/index.html.
curl -I https://aisafework.netlify.app/v2/
# Expect: 200, all headers above present.
```

**Residual risk:** Low. Headers are testable; CI step recommended (see roadmap below).

## OWASP A06 — Vulnerable & outdated components

**Applicability:** LOW. There are **zero JS dependencies**. Zero CSS dependencies beyond the Google Fonts CSS file.

**Mitigations in place:**

- No `package.json`. No `node_modules`. No vendored libraries. No build step.
- Google Fonts CSS is reviewed by Google's own security team; we are at the same risk level as every Google Fonts user. If a higher tier of paranoia is required, fonts can be self-hosted (~80 kB woff2) and the `style-src` + `font-src` directives tightened to `'self'`.

**Verification:**

```bash
# Should output ZERO third-party JS or CSS references.
grep -rE "<script[^>]+src=\"(https?:)?//" v2/
grep -rE "<link[^>]+href=\"(https?:)?//(?!fonts\\.googleapis)" v2/
```

**Residual risk:** Low. Roadmap: optional move to self-hosted fonts for paid surfaces (`pricing.html`, etc.).

## OWASP A08 — Software & data integrity failures

**Applicability:** LOW.

**Mitigations in place:**

- Site source-of-truth is the GitHub repo (`Afrhub/ai-safe-at-work`), branch-protected on `main` (no force-push, linear history required, conversation-resolution required).
- Netlify build picks up from this branch; no out-of-band content injection path.
- `modules.json` is JSON and shape-validated at read time.
- Subresource Integrity (SRI) on Google Fonts: **not feasible** — Google rotates the CSS URLs/contents to deliver browser-specific woff2 subsets. Mitigation: rely on the CSP (`style-src https://fonts.googleapis.com` only) + HTTPS.

**Roadmap:** Self-host fonts on paid surfaces; add SRI to inline `<link>` for the now-static woff2 files.

**Residual risk:** Low.

---

## OWASP A09 — Security logging & monitoring failures

**Applicability:** LOW (nothing to log — no auth, no transactions).

**Mitigations in place:**

- Netlify access logs available to the platform operator on demand.
- `console.warn`/`console.error` paths inside `v2.js` for client-side rate-limit triggers and manifest-load failures. (No PII logged.)
- `.audit/security/incident-log.md` exists for documenting any future incident; integrates with the v1 incident process.

**Residual risk:** Low. If/when paid surfaces add auth, this section needs reworking (auth-event logging, failed-login alerting, etc.).

---

## OWASP A10 — Server-side request forgery (SSRF)

**Applicability:** N/A. No server-side code. No URL-fetching surface.

**Note:** Client-side `fetch()` in `v2.js` is restricted to relative same-origin paths (`fetch('modules.json', { credentials: 'omit', cache: 'default' })`). CSP `connect-src 'self'` enforces this at the browser level.

**Residual risk:** Negligible.
