# Incident log

> **Owner:** Founder · **Effective:** 2026-05-19 · **Reg basis:** ISO/IEC 27001 A.5.24–26 · GDPR Art 33 · EU AI Act Art 73

A chronological log of every information-security, AI, or operational incident — including near-misses. Empty is fine; missing isn't.

## Format

Each incident gets:

- **ID:** `INC-YYYY-NN` (NN sequential within the year)
- **Detected:** datetime
- **Resolved:** datetime
- **Category:** data-leak / bad-output / successful-attack / vendor-incident / near-miss / other
- **Severity:** low / medium / high / critical
- **Reported to authority?** Yes/No + which / when
- **Reported to subjects?** Yes/No + how / when
- **Summary:** plain English
- **Root cause:** identified or hypothesised
- **Actions taken:** chronological
- **Lessons captured:** what changes going forward
- **Linked artefacts:** files / commits / messages / screenshots

## Annual review

Each year:

1. Read every incident from the prior 12 months.
2. Identify patterns (recurring causes, recurring affected systems).
3. Update the risk register if a pattern indicates an unmitigated risk.
4. Write a one-page annual incident summary in `annual-review-YYYY.md`.
